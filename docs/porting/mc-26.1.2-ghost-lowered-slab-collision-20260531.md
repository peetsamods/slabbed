## Current Finding

The latest recorder evidence does **not** prove a simple client state-shape collision mismatch for named lowered slabs.

The lowered-slab state rows contain `4171` cursor rows grouped into four named lowered slab shape groups. Every group has outline, interaction/raycast replay bounds, and collision bounds aligned:

```text

This disproves only the narrow hypothesis that the sampled client `BlockState.getCollisionShape(...)` for named lowered slabs is empty or still dy0 while outline/raycast are lowered.

## Physical Collision Gap

The current recorder still does not prove or disprove the live pass-through symptom because it did not sample the movement collision iterator.

The 26.1.2 movement collision path uses `BlockCollisions`, which asks `CollisionContext.getCollisionShape(state, world, pos)`, moves that shape into world coordinates, then yields it only if it intersects the entity/query shape. The next live proof must therefore compare:

- target lowered slab outline/interaction/raycast/collision bounds
- player AABB and movement-expanded collision query box
- `Level.getBlockCollisions(player, queryBox)` returned shapes
- whether the target lowered slab's world collision bounds are present in those returned shapes

`LiveCursorIntentRecorder` now adds:

```

Marker meaning:

```text
LIVE_COLLISION_ITERATOR_TARGET_MISS
```

means the final target is a lawful lowered lane, its target collision world bounds intersect the player collision query box, but the client movement collision iterator did not return any shape intersecting those target bounds.

That is the next expected RED for a client-side physical collision source miss. If this marker stays zero during a pass-through repro, the next authority layer is likely server-side collision/attachment truth rather than the client state-shape or client collision iterator.
